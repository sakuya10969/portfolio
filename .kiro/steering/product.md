# プロダクト方針

## プロダクト概要

このサイトは、エンジニアとしての技術力・設計力・制作実績・学習姿勢を外部に伝えるためのポートフォリオサイトである。
単なるプロフィール紹介ではなく、各制作物の背景・技術選定理由・設計思想・工夫・課題・改善余地まで含めて伝える構成にする。

## ターゲットユーザー

- 採用担当者（エンジニア採用に関わる人）
- エンジニア（技術的な関心から閲覧する同業者）
- 技術に関心のある第三者（学生、非エンジニアの技術関係者）

閲覧者が初見で「何をやっている人間か」を把握でき、短時間で主要な制作物と技術スタックに到達できる導線を最優先する。

## コアページ一覧

| ページ | 役割 |
|---|---|
| Home | ファーストビューで人物像と代表的な制作物を提示する。全体への導線起点 |
| About | 経歴、価値観、技術的な関心領域を伝える |
| Projects | 制作物一覧。カテゴリ・技術でのフィルタリングを提供する |
| Project Detail | 個別制作物の詳細。背景から改善予定まで網羅する |
| Skills / Tech Stack | 技術スタックを分類・習熟度とともに提示する |
| Experience | 職歴・学歴・活動歴を時系列で整理する |
| Contact | 問い合わせフォーム。簡潔に連絡手段を提供する |

## 各ページで伝えるべき情報

### Home
- 名前・肩書き・一言紹介
- 代表的なプロジェクト（2〜3件）のハイライト
- 技術スタックの概要
- 各セクションへの導線

### About
- 自己紹介（経歴の要約）
- エンジニアとしての価値観・関心領域
- SNS / GitHub などの外部リンク

### Projects
- プロジェクト一覧（カード形式）
- カテゴリフィルタ、技術フィルタ
- 各カードからDetail への遷移

### Project Detail
各プロジェクトで以下を必ず扱う：
- タイトル
- 概要
- 背景 / なぜ作ったか
- 技術スタック
- 設計 / アーキテクチャの要点
- 工夫した点
- 苦労した点
- 今後の改善予定
- GitHub / Demo / 関連ドキュメントへのリンク

### Skills / Tech Stack
- 技術をカテゴリ別に分類（言語、フレームワーク、インフラ等）
- 各技術の習熟度や使用経験の目安
- プロジェクトとの関連付け

### Experience
- 職歴・学歴・活動歴を時系列で表示
- 各項目に期間、役割、概要を記載

### Contact
- 問い合わせフォーム（名前、メール、件名、本文）
- バリデーション付き
- 送信完了のフィードバック

## UI/UX 方針

- 初見で役割が伝わること。ファーストビューで「エンジニアのポートフォリオ」と即座に認識できる
- 視認性と読みやすさを最優先する。装飾よりも情報の明瞭さを重視する
- 3クリック以内で主要プロジェクト詳細に到達できること
- レスポンシブ対応を前提とする。モバイルファーストで設計し、デスクトップで拡張する
- ダークモードをサポートする（light / dark / system）
- アニメーションは情報理解を補助する範囲で使う。演出目的の過剰なアニメーションは避ける

## 見せ方の優先順位

1. 制作物（Projects）— 何を作ったかが最も重要
2. 技術スタック（Skills）— 何ができるかを裏付ける
3. 経歴（Experience）— どういう文脈で活動してきたか
4. 人物像（About）— 価値観と関心領域
5. 連絡手段（Contact）— アクションへの導線

## アニメーションの役割

アニメーションは「情報の出現タイミングを制御し、視線誘導を補助する」ために使う。
具体的な利用箇所は tech.md のアニメーション方針に準拠する。
「動くから楽しい」ではなく「動くことで理解しやすい」を基準にする。

## 情報設計の基本方針

- 各ページは単一の責務を持つ。1ページに複数の役割を詰め込まない
- 情報の階層は浅く保つ。深いネストは避ける
- 重要な情報ほど上部・左側に配置する（F字パターンを意識）
- 繰り返し構造（カード、リスト）は統一されたフォーマットで表示する
- 空状態・ローディング状態・エラー状態を必ず設計する
